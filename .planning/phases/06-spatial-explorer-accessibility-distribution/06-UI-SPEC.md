# Phase 6 — UI Design Contract

> Visual and interaction contract for the Spatial Explorer, accessibility retrofit, high contrast mode, and distribution readiness. This phase adds the signature spatial discovery view, makes the entire application keyboard-navigable and screen-reader-accessible, and prepares the app for macOS distribution. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

### Spatial Map Dimension Color Scales

Each dimension axis uses a distinct color scale. The dominant axis (highest slider weight) drives node fill color.

| Dimension | Scale | Rationale |
|-----------|-------|-----------|
| Key | Circle-of-fifths hue rotation: C=0deg (red) through G=210deg (cyan), wrapping 360deg. Saturation 70%, Lightness 55%. | Musically adjacent keys get adjacent hues. Standard music theory mapping. |
| Tempo | Sequential gradient: 40 BPM = hsl(220, 60%, 50%) (cool blue) to 200 BPM = hsl(0, 80%, 55%) (warm red). | Slow = cool, fast = hot. Intuitive thermal metaphor. |
| Date | Sequential gradient: oldest = hsl(220, 10%, 35%) (faded grey-blue) to newest = hsl(28, 90%, 58%) (accent amber). | Recent jams glow warm; old jams fade into the background. |
| Instruments | Categorical palette (up to 8 distinct hues): hsl(28, 70%, 55%), hsl(180, 60%, 45%), hsl(280, 50%, 55%), hsl(45, 80%, 50%), hsl(340, 60%, 50%), hsl(150, 50%, 45%), hsl(200, 70%, 50%), hsl(60, 60%, 50%). | Distinct hues for the most-used instruments. Overflow uses muted foreground. |
| Collaborators | Same categorical palette as instruments (shared pool). | Collaborator count is typically small. Reuse avoids palette bloat. |

**Color legend:** Always visible in the control panel. Shows the active dimension name, a gradient or swatch bar, and min/max labels. Updates dynamically when the dominant axis changes.

## Focal Point

**Primary visual anchor:** The spatial map canvas. Nodes in dimension-mapped colors float against the dark background. The canvas fills the viewport, making the spatial map feel expansive and explorable — like a musical galaxy.

**Secondary anchor:** The dimension control panel on the right. Sliders with labeled weights let the musician steer the clustering. The active color legend reinforces what the colors mean.

### Spatial Map Node States

**Default (circle):**
- 12px radius circle
- Fill: dimension-mapped color (driven by dominant axis)
- Stroke: 1px node-stroke-default (#3D4556)
- No label visible

**Hover (expanded):**
- Expands to 48px radius over 150ms ease-out
- Shows: mini waveform thumbnail (rendered to canvas, 80x32px area within node), jam name (label/12px, white, truncated at 16 chars), key + BPM badges (label/12px)
- Stroke: 2px node-stroke-selected (white)
- Drop shadow: 0 4px 12px rgba(0,0,0,0.4)
- Nearby nodes gently push away (force simulation charge adjustment)

**Keyboard focused:**
- Same size as default (12px radius) unless Enter pressed (then expands)
- Stroke: 3px accent (#E8863A)
- ARIA live region announces: "{jam name}, {duration}, {key} at {BPM} BPM"
- Arrow keys move focus to nearest neighbor in that direction
- Enter: expands node and shows detail. Second Enter or click: navigates to jam detail view.

**Selected (clicked once):**
- Expanded to 48px
- Stroke: 2px white
- Panel or tooltip anchored to node showing full metadata summary
- Second click: navigates to jam detail view (D-02)

**High contrast mode:**
- All nodes show text labels (jam name) at all times, not just on hover
- Node outlines 3px with high contrast stroke color
- Filled circles become outlined circles with dimension-colored borders

### Screen Reader Strategy (per D-08)

Announcements are concise — name and essentials only. Drill-down provides detail.

- Jam card: "{jam name}, {duration}" — Enter to hear key, BPM, tags
- Spatial node: "{jam name}, {duration}, {key} at {BPM} BPM" — Enter for full detail
- Transport: "Playing {jam name} at {time} of {duration}" / "Paused"
- Tag chip: "{tag name}, tag, {position} of {total}" — Delete to remove

## Phase 6 Specific Notes

1. **Spatial map is the signature feature.** The force-directed graph should feel magical and explorable. Performance target: smooth 60fps pan/zoom with up to 500 nodes. react-force-graph uses Canvas rendering which handles this well.

2. **Accessibility is a retrofit, not a rewrite.** Existing components keep their visual appearance. The retrofit adds: skip link, landmark roles, roving tabindex, ARIA labels, keyboard handlers, and live regions. shadcn/radix components already have baseline ARIA — verify and augment, don't replace.

3. **High contrast uses CSS media query only.** No in-app toggle. `@media (prefers-contrast: more)` overrides CSS custom properties. This is a CSS-only change with no JavaScript runtime cost.

4. **Dimension blending is the UX innovation.** Multiple sliders controlling force simulation weights let musicians ask spatial questions about their library. The implementation uses d3 scales to map dimension values to force strengths, then react-force-graph renders the result.

5. **Node interaction is two-step.** First click/Enter selects and expands. Second click/Enter navigates to jam detail. This prevents accidental navigation when exploring the map.

6. **Distribution is non-UI.** Code signing (INFRA-14), notarization, and GitHub Actions CI/CD are backend/infra concerns with no frontend visual impact beyond the first-launch dialog (INFRA-13) and settings toggle.

7. **Canvas accessibility requires extra care.** react-force-graph renders to HTML5 Canvas, which is opaque to screen readers. The accessibility layer uses a hidden DOM overlay with ARIA-annotated elements that mirror the canvas nodes. Arrow key navigation uses spatial proximity calculations, not DOM order.
