    **Create `crates/wallflower-core/src/watcher/mod.rs`:**
    Per D-09 (watch one configured folder), D-11 (5-second debounce):

    - Define `WatcherHandle` struct that holds a `notify::RecommendedWatcher` and a `JoinHandle` for the debounce thread, plus an `Arc<AtomicBool>` for active status tracking.
    - `pub fn start_watcher(watch_path: PathBuf, db_path: PathBuf, storage_dir: PathBuf) -> Result<WatcherHandle>`:
      1. Create `std::sync::mpsc::channel()` for notify events.
      2. Create `RecommendedWatcher` with `notify::Config::default()`, send events through channel.
      3. Call `watcher.watch(&watch_path, RecursiveMode::Recursive)`.
      4. Spawn a debounce thread that:
         - Maintains a `HashMap<PathBuf, Instant>` of pending files.
         - Reads events from channel via `try_recv()` in a loop with 500ms sleep.
         - On `EventKind::Create(_) | EventKind::Modify(_)`, insert/update the path in pending if `import::is_audio_file(&path)` returns true.
         - Check pending entries: if `now - last_modified >= Duration::from_secs(5)`, open a new Database connection using `db_path`, load config, call `import::import_file(&db.conn, &storage_dir, &path)`, log the result with `tracing::info!`, remove from pending.
      5. Set the `AtomicBool` active flag to true.
      6. Return `WatcherHandle` with the watcher, join handle, and active flag.
    - `impl WatcherHandle`:
      - `pub fn is_active(&self) -> bool` — reads the AtomicBool.
      - `pub fn stop(self)` — drops the watcher (stops watching), signals the debounce thread to exit via a separate AtomicBool stop flag, joins the thread.
    - Add test: `test_watcher_detects_new_file` — create a temp dir, start watcher on it, write a small WAV file into it, wait 6 seconds, verify the file was imported into a test database.

    **Create `crates/wallflower-core/src/device/mod.rs`:**
    Per D-02 (USB recorder detection shows import prompt), D-09 (auto-detect mounted USB volumes):

    Use a pragmatic approach: scan `/Volumes/` for mounted volumes that look like audio recorders.
    - Define `DeviceInfo` struct:
      ```rust
      #[derive(Debug, Clone, Serialize, Deserialize)]
      #[serde(rename_all = "camelCase")]
      pub struct DeviceInfo {
          pub name: String,
          pub mount_point: String,
          pub files: Vec<String>,
      }
      ```
    - `pub fn detect_devices() -> Vec<DeviceInfo>`:
      1. List entries in `/Volumes/`.
      2. For each volume, check if it looks like a Zoom F3 or similar recorder by looking for characteristic directory structures:
         - Zoom F3: has a `ZOOM0001/` or similar numbered folder containing WAV files, or has `.WAV` files in root or `STEREO/` subdirectory.
         - Generic detection: any removable volume with audio files (.wav, .flac, .mp3) in it.
      3. For each detected device, collect the list of audio file paths.
      4. Return `Vec<DeviceInfo>` with device name (volume name), mount point, and audio file list.
    - `pub fn is_zoom_recorder(mount_point: &Path) -> bool` — check for Zoom-specific directory patterns (e.g., folders named `ZOOM0001`, `ZOOM0002`, etc., or `STEREO/` folder with WAV files).
    - `pub fn find_audio_files_on_device(mount_point: &Path) -> Vec<PathBuf>` — recursively find all audio files on the volume, limited to 2 directory levels deep (avoid scanning entire USB drives).
    - Add test: `test_detect_devices_returns_empty_on_no_devices` — verify detect_devices returns empty vec when no removable volumes exist (or in CI).
    - Add test: `test_is_zoom_recorder_with_structure` — create a temp dir mimicking Zoom F3 structure, verify detection.
