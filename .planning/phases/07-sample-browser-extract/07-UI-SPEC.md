# Phase 7 -- UI Design Contract

> Visual and interaction contract for the Sample Browser and Extract feature in the Explore tab. Replaces the spatial explorer placeholder (removed in Phase 6) with a DAW-style sample browser for searching, filtering, previewing, and exporting bookmarks, loops, and sections across all recordings. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (customized preset, established Phase 2) |
| Preset | custom -- dark base, warm accent tokens, extended border-radius |
| Component library | base-ui (via shadcn base-nova style) + custom Tailwind components |
| Icon library | lucide-react |
| Font | Plus Jakarta Sans (400, 600) via @fontsource/plus-jakarta-sans |

**Phase 7 continuity:** No changes to the design system foundation. Phase 7 replaces the Explore tab placeholder with: (1) a sidebar filter panel, (2) a sortable sample table, (3) a bottom preview panel with waveform, and (4) export actions. All use established tokens and shadcn components.

## Focal Point

**Primary visual anchor:** The sample table. Dense, scannable rows of samples with type badges and musical metadata. The table fills the majority of viewport width and all available height. Musicians spend most time here scanning and clicking play.

**Secondary anchor:** The bottom preview panel. Appears when a sample is selected. Shows the full waveform with scrubbing, key/BPM metadata, and export actions. Same spatial position and visual weight as the StemMixer from Phase 5.

### Preview Panel Detail

- Position: Fixed at bottom of Explore tab, above transport bar
- Height: 220px
- Background: secondary (#1D2129)
- Top border: 1px solid --border
- Slides up when a sample is selected (200ms ease-out transition, matches Sheet animation)
- Three-section horizontal layout:
  1. **Waveform area** (flex, takes remaining width): Reuses WaveformDetail component. Peaks loaded from backend. Interactive scrubbing. No bookmark regions overlay (preview shows a single sample). Height: 120px within the panel.
  2. **Metadata row** (below waveform): Sample name (body/14px/semibold), key badge, BPM badge, duration. Single horizontal row, md (16px) padding.
  3. **Action row** (below metadata): "Export Audio" button (accent filled, primary CTA), "Export Stems" button (ghost/secondary), "Go to Jam" link button (ghost, accent text). Right-aligned. md (16px) padding.

### Preview Panel -- Waveform Considerations

- WaveformDetail is reused but with simplified props: no bookmarks array, no sections overlay, no loops overlay. The preview shows only the sample's time range.
- Audio URL: constructed from the source jam's audio file, with the sample's startSeconds/endSeconds defining the visible range.
- If the sample is short (under 5 seconds), waveform zoom defaults to fit the full sample. No horizontal scroll needed.

## Phase 7 Specific Notes

1. **DAW browser mental model.** The layout mirrors Ableton's browser: filters on the left, results in the center, preview at the bottom. Musicians already know this pattern. Do not innovate on the layout -- innovate on the content (cross-jam sample aggregation with musical metadata).

2. **Three sample types in one list.** Bookmarks (user-created), loops (AI-detected), and sections (AI-detected) appear together in a flat list with type badges for differentiation. The Type filter in the sidebar lets users narrow to one type. This is per D-02 from CONTEXT.md.

3. **Preview panel reuses WaveformDetail.** The same wavesurfer-based waveform component from jam detail view. Peaks are pre-computed server-side. No new waveform rendering code needed -- just new composition. The preview panel shows a single sample's time range, not the full recording.

4. **Export without bookmarking.** Per D-16, sections and loops can be exported directly. The export pipeline uses jam ID + time range, which all three types provide. This removes the friction of "create a bookmark first, then export."

5. **Sidebar vs FilterBar.** The Explore tab deliberately uses a sidebar layout (D-11) to differentiate from Library's inline chip-based FilterBar. Both filter UIs serve different workflows: Library's FilterBar is for quick jam filtering inline; Explore's sidebar is for deep sample mining with persistent controls.

6. **Collapsible sidebar.** Per D-14, the sidebar can collapse to maximize table space. The collapsed state shows a filter icon with active filter count. This is important for musicians who want maximum scannable area after setting their filters.

7. **Play triggers preview.** Per D-10, clicking a row's play button both selects the row AND starts playback in the preview panel. This is a single-action pattern -- no "select then play" two-step.

8. **Cross-jam API.** The backend needs a new endpoint that aggregates bookmarks, sections, and loops across all jams with filtering support. This is a backend concern, but the UI contract assumes this endpoint returns a unified list with a type discriminator.

9. **Source jam navigation.** Clicking the source jam name in a table row navigates to the Library tab with that jam's detail view open. This bridges the Explore and Library experiences.

10. **No destructive actions.** Phase 7 has no delete or destructive operations. Export only creates new files. The sample browser is read-only with respect to the source data.
