<objective>
Build the Rust backend for cross-jam sample aggregation, filtered querying, and direct sample export. This provides the data layer that the frontend sample browser (Plans 02-04) consumes.

Purpose: The sample browser needs to display bookmarks, sections, and loops from ALL jams in a single sortable/filterable list. The existing API only supports per-jam queries. This plan creates: (1) a UNION-based SQL query aggregating all three sample types, (2) Tauri commands to expose the query and filter options, (3) export commands that accept jam_id + time range directly (per D-16, D-17) so sections and loops can be exported without creating bookmarks.

Output: Rust types, DB query functions, Tauri commands -- all testable via `cargo test --workspace`.
</objective>

```rust
/// A unified sample record representing a bookmark, section, or loop.
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct SampleRecord {
    pub id: String,
    pub sample_type: String,        // "bookmark", "section", or "loop"
    pub jam_id: String,
    pub name: String,
    pub start_seconds: f64,
    pub end_seconds: f64,
    pub color: Option<String>,      // bookmark color, null for sections/loops
    pub repeat_count: Option<i32>,  // loop repeat count
    pub evolving: bool,             // loop evolving flag
    pub source_jam_name: String,
    pub jam_imported_at: String,
    pub key_display: Option<String>,  // "C minor" or null
    pub tempo_bpm: Option<f64>,
    pub duration_seconds: f64,
    pub notes: Option<String>,      // bookmark notes, jam notes for others
}

/// Filter criteria for the sample browser.
#[derive(Debug, Clone, Default, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct SampleFilter {
    pub query: Option<String>,
    pub types: Option<Vec<String>>,     // ["bookmark", "section", "loop"]
    pub keys: Option<Vec<String>>,
    pub tempo_min: Option<f64>,
    pub tempo_max: Option<f64>,
    pub duration_min: Option<f64>,
    pub duration_max: Option<f64>,
    pub source_jam_id: Option<String>,
    pub tags: Option<Vec<String>>,
}

/// Available filter option values for the sample browser sidebar.
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct SampleFilterOptions {
    pub keys: Vec<String>,
    pub tags: Vec<String>,
    pub jams: Vec<(String, String)>,  // (id, original_filename)
    pub tempo_min: f64,
    pub tempo_max: f64,
    pub duration_min: f64,
    pub duration_max: f64,
}
```

**In `mod.rs`**, add two new public functions after `list_jams_spatial()`:

1. **`get_all_samples(conn: &Connection, filter: &SampleFilter) -> Result<Vec<SampleRecord>>`**

Build a UNION ALL query across `bookmarks`, `jam_sections`, and `jam_loops`, each JOINing `jams`, LEFT JOINing `jam_key` and `jam_tempo`. The three SELECT branches must each produce the same column set:
- `id, sample_type, jam_id, name, start_seconds, end_seconds, color, repeat_count, evolving, source_jam_name, jam_imported_at, key_display, tempo_bpm, duration_seconds, notes`

For bookmarks: `b.color`, `NULL as repeat_count`, `0 as evolving`, `b.notes`
For sections: `NULL as color`, `NULL as repeat_count`, `0 as evolving`, `j.notes`  
For loops: `NULL as color`, `l.repeat_count`, `l.evolving`, `j.notes`

Wrap the UNION ALL in a CTE: `WITH all_samples AS (... UNION ALL ...) SELECT * FROM all_samples WHERE ...`

Apply filter conditions using the same dynamic `param_idx` + `conditions` + `param_values` pattern from `search_jams()`:
- `filter.types`: `sample_type IN (?, ?, ...)` 
- `filter.keys`: `key_display IN (?, ...)`
- `filter.tempo_min`/`filter.tempo_max`: `tempo_bpm >= ? AND tempo_bpm <= ?`
- `filter.duration_min`/`filter.duration_max`: `duration_seconds >= ? AND duration_seconds <= ?`
- `filter.source_jam_id`: `jam_id = ?`
- `filter.tags`: `EXISTS (SELECT 1 FROM jam_tags WHERE jam_tags.jam_id = all_samples.jam_id AND jam_tags.tag IN (?, ...))`
- `filter.query`: `(name LIKE ? OR source_jam_name LIKE ? OR notes LIKE ?)` with `%query%` pattern (reuse the same `?` param index for each LIKE, binding the pattern value 3 times)

Add `ORDER BY jam_imported_at DESC` as default sort.
Add `LIMIT 5000` as a safety bound.

Map each row to `SampleRecord` using:

Reuse existing `get_distinct_keys(conn)` and `get_tempo_range(conn)` and `list_all_tags(conn)`. Add:

Return `SampleFilterOptions { keys, tags, jams, tempo_min, tempo_max, duration_min, duration_max }`.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | db::get_all_samples SQL query | mitigate | All filter values bound via `rusqlite::params![]` / positional `?N` parameters. No string interpolation of user values. Same pattern as existing search_jams(). |
| T-07-02 | Tampering | export_sample_audio file path | mitigate | Reuse `export::sanitize::resolve_export_path()` which sanitizes filenames and prevents directory traversal. Already validated in Phase 5. |
| T-07-03 | Denial of Service | get_all_samples unbounded results | mitigate | Add `LIMIT 5000` to the UNION query. Desktop app with local data, risk is minimal, but bounds prevent pathological cases. |
| T-07-04 | Information Disclosure | sample notes field | accept | Notes are user's own data in a single-user local app. No cross-user data access possible. |
